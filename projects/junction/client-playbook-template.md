## Executive Summary

This playbook defines how Junction delivers exceptional results for every client through our hybrid AI-human model. Account Managers (AMs) oversee a portfolio of 4 clients, providing strategic judgment and relationship management while AI agents handle 90% of execution.

**The Junction Promise:** Expert strategy, AI-powered execution, 10x the output at a fraction of traditional agency cost.

|--------|--------|---------------|
| **Deliverables on time** | 25% | 5=100%, 3=80%, 1=<60% |
| **Performance vs goals** | 25% | 5=Exceeding, 3=On track, 1=Behind |

| **AM relationship quality** | 20% | 5=Strong, 3=Neutral, 1=Strained |

**Health Score Interpretation:**
- **4.5-5.0 (Green):** Healthy, growth opportunity
- **3.5-4.4 (Yellow):** Monitor closely, proactive outreach
- **2.5-3.4 (Orange):** Intervention needed, escalate to Tom

## Part 7: AI Agent Handoff Protocol

> *"What breaks first at scale is agent continuity. This section is non-negotiable."* — JARVIS

### Why This Matters

When multiple AI agents touch the same client — whether across sessions, tasks, or escalations — context loss creates:
- Inconsistent brand voice
- Repeated questions to clients
- Contradictory outputs
- Dropped balls on ongoing work
- Erosion of the "one team" experience

**The goal:** Any agent can pick up any client's work mid-stream and continue seamlessly.

---

### Client Context Architecture

Every client has a **canonical context stack** — the single source of truth that all agents read from and write to.

```
clients/
└── [client-slug]/
    ├── STATE.json           # Fast-tier: current status, active work, blockers
    ├── CONTEXT.md           # Slow-tier: who they are, what matters, preferences
    ├── VOICE.md             # Brand voice guide, tone, terminology
    ├── memory/
    │   ├── YYYY-MM-DD.md    # Daily work logs
    │   └── decisions.md     # Key decisions and rationale
    ├── content/
    │   ├── queue/           # Drafts awaiting review
    │   ├── approved/        # Ready to publish
    │   └── published/       # Archive
    └── assets/
        ├── brand/           # Logos, colors, fonts
        └── reference/       # Competitor examples, inspiration
```

---

### Agent Session Protocol

#### Starting a Session (Mandatory Steps)

```
1. READ STATE.json
   └─ Understand current status, blockers, active work

2. CHECK memory/[today].md and memory/[yesterday].md
   └─ What happened recently? Any handoff notes?

3. IF first time on this client (or >7 days since last touch):
   └─ READ CONTEXT.md fully
   └─ READ VOICE.md

4. IDENTIFY your task
   └─ What am I here to do?
   └─ What's already been done toward this?

5. BEGIN work with full context
```

#### Ending a Session (Mandatory Steps)

```
1. UPDATE STATE.json
   └─ pendingTasks: Add/update/complete as needed
   └─ lastUpdated: Current timestamp
   └─ lastAgent: Your agent ID
   └─ blockers: Add any new blockers
   └─ recentDecisions: Log any decisions made

2. WRITE to memory/[today].md
   └─ What you completed
   └─ Any client interactions
   └─ Decisions made and why
   └─ Explicit handoff notes for next agent

3. SAVE all work products
   └─ Drafts → content/queue/
   └─ Include metadata (task ID, status, notes)

4. FLAG if escalation needed
   └─ Update STATE.json flags
   └─ Notify AM via appropriate channel
```

### Context Sync Rules

#### What Gets Written Where

| Information Type | Location | Who Writes |
|------------------|----------|------------|
| Current task status | STATE.json | Every agent, every session |
| Work completed today | memory/YYYY-MM-DD.md | Every agent |
| Client preferences (stable) | CONTEXT.md | AM only (agents can propose) |
| Brand voice rules | VOICE.md | AM only |
| Strategic decisions | memory/decisions.md | AM or escalated agent |
| Content drafts | content/queue/ | Content agents |
| Performance data | analytics/ | Analytics agents |

#### Conflict Resolution

**If two agents update STATE.json simultaneously:**
1. Later timestamp wins for `lastUpdated` and `lastAgent`
2. `pendingTasks` arrays merge (no duplicates by ID)
3. `blockers` arrays merge (deduplicate by description)
4. Conflicts in other fields → flag for AM review

**If context seems wrong or outdated:**
1. Agent flags in STATE.json: `"contextStale": true`
2. Note specific concern in daily memory file
3. AM reviews on next check-in

### Failure Modes & Recovery

#### "Agent has no idea what's happening"
**Symptom:** Agent asks client questions already answered  
**Cause:** Didn't read context files  
**Fix:** Enforce session protocol; add pre-flight checklist to agent prompts

#### "Agent contradicts previous work"
**Symptom:** New content conflicts with established voice/decisions  
**Cause:** VOICE.md or decisions.md not read  
**Fix:** Add explicit "check before creating" step; AM review catches

#### "Work gets duplicated"
**Symptom:** Two agents create same deliverable  
**Cause:** STATE.json not updated; pendingTasks out of sync  
**Fix:** Enforce "claim task" step — update STATE.json before starting

#### "Context file is a mess"
**Symptom:** Files too long, outdated info, contradictions  
**Cause:** No regular maintenance  
**Fix:** Weekly AM review; monthly Ops cleanup; hard size limits

#### "Agent can't find previous work"
**Symptom:** Agent recreates something that exists  
**Cause:** Poor file organization or naming  
**Fix:** Strict folder structure; task ID in all filenames; search before create

### Agent Prompt Requirements

Every agent prompt for client work MUST include:

```
## Context Loading (Required)
Before starting any work on {client}:
1. Read clients/{client}/STATE.json
2. Read clients/{client}/memory/[today].md and [yesterday].md
3. If unfamiliar with client: Read CONTEXT.md and VOICE.md
4. Check content/queue/ for related in-progress work

## Context Saving (Required)
Before ending session:
1. Update STATE.json with current status
2. Append to memory/[today].md with work summary + handoff notes
3. Save all outputs to appropriate content/ folder
4. Flag any blockers or escalations

Failure to follow context protocol = broken client experience.
```
